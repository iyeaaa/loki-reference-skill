# 온보딩 Step 4 (캠페인 실행) UI/UX 재설계 문서

## 1. 개요

### 1.1 목적
온보딩 Step 4에서 사용자가 **리드 목록**과 **이메일 내용**을 모두 확인하고, 필요시 수정한 후 캠페인을 실행할 수 있는 최적의 UI/UX 설계

### 1.2 현재 위치
```
온보딩 플로우:
Step 1: 회사 정보 입력
Step 2: 이메일 연동
Step 3: 바이어 찾고 이메일 생성 (로딩/진행 상황)
Step 4: 캠페인 실행 ⬅️ 개선 대상
```

### 2.3 현재 문제점

| 문제 | 현재 상태 | 영향 |
|------|----------|------|
| 리드 일부만 표시 | 최대 4개만 미리보기 | 전체 리드 확인 불가 |
| 리드 수정 불가 | 제외/수정 기능 없음 | 원치 않는 리드에 발송될 수 있음 |
| 이메일 내용 없음 | 이메일 수만 표시 (90개) | 어떤 이메일이 발송되는지 확인 불가 |
| 이메일 수정 불가 | 편집 기능 없음 | 오타/수정 필요 시 대응 불가 |

## 3. 제안 UI/UX 설계

### 3.1 디자인 원칙

1. **완전한 투명성**: 발송 전 모든 정보를 확인할 수 있어야 함
2. **최종 수정 기회**: 실행 직전 리드 제외, 이메일 수정 가능
3. **심플한 UX**: 온보딩 단계이므로 복잡하지 않게
4. **모바일 친화적**: 좁은 화면에서도 사용 가능

### 3.2 새로운 레이아웃: 탭 기반 리뷰 UI

### 6.2 상태 관리
```typescript
// StepConfirmation 상태
interface ConfirmationState {
  // 탭 상태
  activeTab: 'leads' | 'emails' | 'execute';

  // 리드 상태
  allLeads: Lead[];
  selectedLeadIds: string[];
  leadSearchQuery: string;

  // 이메일 상태
  steps: EmailStep[];

  // 실행 상태
  isExecuting: boolean;
  executionProgress: number;
  executionComplete: boolean;
  executionError: string | null;

// Lead 타입
interface Lead {
  id: string;
  companyName: string;
  email?: string;
  country?: string;
  contactName?: string;
}

// EmailStep 타입
interface EmailStep {
  id: string;
  stepOrder: number;
  delayDays: number;
  emailSubject: string;
  emailBodyText: string;
  emailBodyHtml?: string;
}
```

## 7. API 요구사항

### 7.1 기존 API 활용

| API | 용도 | 훅/서비스 |
|-----|------|----------|
| 리드 목록 조회 | 선택된 리드 정보 | `GET /api/v1/leads?ids=...` |
| 시퀀스 스텝 조회 | 이메일 스텝 목록 | `useSequenceSteps(sequenceId)` |
| 스텝 수정 | 이메일 제목/본문 수정 | `useUpdateSequenceStep()` |
| 캠페인 실행 | 리드 등록 + 활성화 | `POST /enrollments/bulk-with-scheduling` |

### 7.2 현재 StepConfirmation에서 사용 중인 API
```typescript
// 리드 조회 (현재 직접 apiFetch 사용)
const fetchLeads = async () => {
  const response = await apiFetch<{ data: Lead[] }>(
    `/api/v1/leads?ids=${selectedLeadIds.join(",")}`
  );
};

// 캠페인 실행
await apiFetch(`/api/v1/admin/sequences/${sequenceId}/enrollments/bulk-with-scheduling`, {
  method: "POST",
  body: JSON.stringify({ leadIds, userEmailAccountId, enrolledBy }),
});

// 캠페인 활성화
await apiFetch(`/api/v1/sequences/${sequenceId}/activate-step-based`, {
  method: "POST",
});
```

### 7.3 추가 필요 API
```typescript
// 시퀀스 스텝 조회 (이메일 내용 확인용)
useSequenceSteps(sequenceId: string)

// 스텝 수정 (이메일 수정용)
useUpdateSequenceStep(sequenceId: string)
```

**참고**: 위 API들은 이미 구현되어 있음 (`admin/src/lib/api/hooks/sequences.ts`)

## 8. 온보딩 특화 고려사항

### 8.1 온보딩 vs 일반 캠페인 차이

| 구분 | 온보딩 Step 4 | 일반 캠페인 Step 3 |
|------|--------------|-------------------|
| **사용자 수준** | 첫 사용자 (초보) | 경험 있는 사용자 |
| **UI 복잡도** | 심플하게 | 기능 풍부하게 |
| **편집 범위** | 제목/본문만 | 발송 시간, 첨부파일 등 |
| **가이드** | 단계별 안내 필요 | 최소한의 안내 |

### 8.2 온보딩에 맞는 단순화

1. **3개 탭으로 제한**: 리드 → 이메일 → 실행
2. **필터 최소화**: 검색만 제공 (국가, 업종 필터 생략)
3. **편집 기능 제한**: 제목/본문만 수정 (발송 시간, 첨부파일 생략)
4. **가이드 텍스트**: 각 탭에 간단한 설명 추가

### 8.3 토스/카카오 스타일 유지

현재 StepConfirmation은 토스 스타일의 깔끔한 UI를 사용 중:
- 둥근 카드 (`rounded-2xl`)
- 그라데이션 배경 (`bg-gradient-to-br`)
- 심플한 아이콘
- 부드러운 색상 (blue-50, green-50, purple-50)

**유지해야 할 요소:**
- 카드 기반 레이아웃
- 부드러운 그라데이션
- 명확한 CTA 버튼
- 친근한 문구 ("바로 시작할까요?", "나중에 할게요")

## 12. 결론

### 12.1 핵심 개선 사항

| 현재 | 개선 후 |
|------|--------|
| 리드 4개만 표시 | 전체 리드 목록 + 검색 |
| 이메일 내용 없음 | 스텝별 전체 내용 표시 |
| 수정 불가 | 리드 제외 + 이메일 수정 가능 |
| 단일 화면 | 탭 기반으로 정보 분리 |

### 12.2 기대 효과

1. **투명성 향상**: 발송 전 모든 정보 확인 가능
2. **실수 방지**: 원치 않는 리드 제외, 오타 수정 가능
3. **사용자 신뢰**: "내가 통제하고 있다"는 느낌
4. **온보딩 완성도**: 첫 경험의 만족도 향상

### 12.3 예상 개발 기간

| Phase | 작업 | 기간 |

| Phase 3 | 이메일 확인 탭 | 1-2일 |
| Phase 4 | 통합 테스트 | 1일 |
| **Total** | | **4-5일** |
