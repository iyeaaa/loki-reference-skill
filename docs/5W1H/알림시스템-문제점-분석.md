### 문제점 1: 이중 SSE 연결

| 구분 | Onboarding SSE | Notification SSE |
|------|---------------|------------------|
| 엔드포인트 | `/api/v1/onboarding/workspace/{id}/stream` | `/api/v1/notifications/stream` |
| 채널 | `onboarding:progress:{workspaceId}` | `notifications:{userId}` |
| 사용처 | StepBuyerLoading | NotificationBell |
| 데이터 소스 | Redis PubSub 직접 | DB → Redis PubSub |
| 지연 | 실시간 | DB 저장 후 (5-50ms 추가) |

**영향**: 같은 진행 상황을 두 개의 SSE로 중복 전송하며, NotificationBell은 DB를 거치므로 지연 발생

## 3. Fake Progress 문제점

### 현재 동작

```typescript
// fake-progress.ts

// 설정값
const maxFakeProgress = 15   // 최대 15%까지만 fake
const minFakeProgress = 5    // 시작은 5%
const duration = 30_000      // 30초에 걸쳐 5% → 15%

// Ease-out Cubic 곡선
// 0초: 5%
// 5초: 12%
// 15초: 14.5%
// 30초: 15%
```

### 문제점 2: Fake Progress 범위 한계

| 문제 | 설명 |
|------|------|
| 최대 15%까지만 | 30초가 지나도 15%에서 멈춤. 실제 progress가 오지 않으면 멈춘 것처럼 보임 |
| 실제 progress 5% 기준 | `realProgress >= 5`일 때만 실제값 사용. 0~4%는 fake 사용 |
| 초기 지연 | SSE 연결 + 첫 이벤트까지 1-3초 소요되는 동안 5%에서 멈춤 |

### 문제점 3: Fake Progress 동기화 이슈

```typescript
// StepBuyerLoading.tsx
const displayProgress = useSharedFakeProgress(
  workspaceId,
  progressPercent,        // SSE에서 오는 실제값
  viewState === "generating",
  { maxFakeProgress: 15 }
)

// NotificationBell.tsx (NotificationItem)
const displayProgress = useSharedFakeProgressReadOnly(
  workspaceId,
  rawProgressPercent,     // DB의 metadata.progressPercent
  isInProgress,
)
```

| 문제 | 설명 |
|------|------|
| 데이터 소스 불일치 | StepBuyerLoading은 SSE 실시간, NotificationBell은 DB에서 조회 |
| workspaceId 의존 | workspaceId가 없으면 fake progress 작동 안 함 |
| 리마운트 시 초기화 | NotificationBell Popover 닫았다 열면 progress 0부터 시작 가능 |

## 4. 알림 업데이트 스로틀링 문제

### notification.service.ts:567-575

```typescript
// 의미있는 변화인지 확인 (5% 이상 변화 또는 phase 변경)
const progressDiff = Math.abs(event.progressPercent - existingProgress)
const phaseChanged = existingPhase !== event.phase
const shouldUpdate = progressDiff >= 5 || phaseChanged

if (!shouldUpdate) {
  // 변화가 미미하면 업데이트 스킵 (SSE 이벤트도 발행하지 않음)
  return existing
}
```

### 문제점 4: 진행률 점프 현상

| 상황 | 결과 |
|------|------|
| progress: 10% → 12% → 14% | DB 업데이트 안됨 (5% 미만) |
| progress: 14% → 15% | 여전히 업데이트 안됨 |
| progress: 10% → 16% | 업데이트 됨 → NotificationBell에서 10% → 16% 점프 |

**영향**: NotificationBell의 진행률이 계단식으로 점프 (부드럽지 않음)

## 5. 타입/상태 불일치 문제

### 문제점 5: 알림 타입 혼재

```typescript
// NotificationBell.tsx의 조건 분기
const isComplete = phase === "complete" || notification.type === "success"
const isError = phase === "error" || notification.type === "error"
const isInProgress = notification.type === "onboarding" && !isComplete && !isError
```

| phase | notification.type | 결과 |
|-------|------------------|------|
| "complete" | "success" | ✅ 완료 표시 |
| "discovery" | "onboarding" | ✅ 진행 중 표시 |
| "error" | "error" | ✅ 에러 표시 |
| "complete" | "onboarding" | ❌ 진행 중으로 잘못 표시 (race condition) |

### 문제점 6: metadata 접근 불안정

```typescript
// NotificationBell.tsx
const metadata = notification.metadata as {
  phase?: string
  progressPercent?: number
  // ...
} | null

const rawProgressPercent = metadata?.progressPercent ?? 0  // null일 때 0
```

**영향**: metadata가 없거나 형식이 다르면 progress가 0으로 표시

## 6. SSE 연결 관리 문제

### 문제점 7: StepBuyerLoading SSE 조건

```typescript
// StepBuyerLoading.tsx:124-128
const shouldEnableSSE =
  !!workspaceId &&
  (viewState === "generating" ||
    onboardingData?.jobStatus === "active" ||
    onboardingData?.jobStatus === "waiting")
```

| 상황 | shouldEnableSSE | 문제 |
|------|-----------------|------|
| 페이지 진입 직후 | `false` (viewState="loading") | SSE 연결 안됨 |
| DB 조회 완료 | `true` (jobStatus 확인 후) | 연결 시작 (1-2초 지연) |
| 완료 후 재진입 | `false` (jobStatus="completed") | 연결 안함 ✅ |

**영향**: 페이지 진입 후 1-2초간 진행률 표시 안됨

### 문제점 8: Notification SSE 재연결

```typescript
// notifications.ts:363-375
if (reconnectAttemptsRef.current < maxReconnectAttempts && enabled) {
  const delay = Math.min(1000 * 2 ** reconnectAttemptsRef.current, 30_000)
  // 지수 백오프: 1s, 2s, 4s, 8s, 16s (최대 30s)
  reconnectTimeoutRef.current = setTimeout(() => {
    reconnectAttemptsRef.current++
    connect()
  }, delay)
} else if (reconnectAttemptsRef.current >= maxReconnectAttempts) {

```

**영향**: 네트워크 불안정 시 최대 30초간 알림 수신 불가

## 7. 데이터 흐름 타임라인 (정상 케이스)

```
시간  │ Worker              │ StepBuyerLoading    │ NotificationBell
──────┼─────────────────────┼─────────────────────┼────────────────────
0ms   │ emitAndSave() 호출  │ SSE 대기 중         │ SSE 대기 중
      │                     │                     │
5ms   │ ├─ Redis PubSub     │                     │
      │ │  publish          │                     │
      │                     │                     │
10ms  │                     │ SSE event 수신      │
      │                     │ progress: 25%       │
      │                     │                     │
15ms  │ └─ DB insert        │                     │
      │    + Redis PubSub   │                     │
      │                     │                     │
50ms  │                     │                     │ SSE event 수신
      │                     │                     │ (DB 통해서 옴)
      │                     │                     │ invalidateQueries
      │                     │                     │
100ms │                     │                     │ React Query refetch
      │                     │                     │ progress: 25%
```

**지연 원인**:
1. DB 저장 시간: ~10-30ms
2. React Query invalidation + refetch: ~50-100ms
3. **총 지연: ~50-150ms**

              └─ DB (비동기, SSE 무관)
